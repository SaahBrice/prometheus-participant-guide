# The 2-minute demo

The video is 25% of your score and the only part of your project every judge is guaranteed to experience fully. Teams treat it as an afterthought; winners treat it as a deliverable.

## A structure that works

- 0:00–0:20 — The problem. A real person, a real pain. "Maria teaches 60 students and spends her evenings grading" beats "education has challenges."
- 0:20–0:40 — Your solution in one sentence, then straight into the screen.
- 0:40–1:30 — The live flow, start to finish, no detours. One complete journey through the product. Show real input and real output.
- 1:30–1:50 — What the AI is doing under the hood. Judges score Creative AI/ML Use partly from this — say explicitly what the model does and why the product needs it.
- 1:50–2:00 — The impact, one line. Who is better off and how.

## Rules learned from watching hundreds of these fail

- Record the screen with real data, not lorem ipsum and "test test 123".
- Script it word for word, then rehearse until it sounds unscripted. Rambling costs you the whole middle section.
- Use a decent microphone or a quiet room. Bad audio reads as a bad product.
- Do not spend 40 seconds on team introductions or slides about the market size. Judges want to see the product working.
- Do not show code in the video unless the code is the product. The repo is where code lives.
- Record the demo flow twice; something always glitches the first time.
- Cut ruthlessly to fit 2:00. Going over signals you couldn't prioritize — the same skill the whole hackathon tests.

## Before you submit

Run the checklist: video is 2 minutes or under and the link is public; the repo is public, has a README with run instructions, and contains no API keys; the prototype actually runs from a fresh clone; the submission form is complete. Then submit hours before the deadline, not minutes — Devpost deadlines do not negotiate.

Good luck. Build one thing well, explain it clearly, and you are in the top ten already.
